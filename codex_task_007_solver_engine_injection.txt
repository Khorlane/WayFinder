WAYFINDER CODEX TASK 007
Solver Engine Injection (Remove Solver Construction from Mapper)

Objective
---------
Complete the next architectural decoupling step by removing direct solver
construction from Mapper.

Currently Mapper still creates the solver internally via:

    func (m *Mapper) solver() *ConstraintSolver

This means Mapper still owns solver construction details.

We want the next cleaner boundary:

- Mapper depends on a solver interface
- concrete solver construction happens outside Mapper
- Mapper coordinates but does not construct the solver implementation

Important
---------
Do NOT change observable behavior.
Do NOT redesign solver algorithms.
Do NOT change spatial semantics.
Do NOT change discovery semantics.
Do NOT change rebuild semantics.

This is a dependency-structure refactor only.

Why
---
Current state is very good:

- solver is extracted
- solver works on explicit solver state
- solver returns explicit rebuild results

But Mapper still constructs the concrete solver.

The next cleaner architecture is:

Mapper → depends on SolverEngine interface
Concrete solver → provided/injected

Scope
-----
Introduce a solver interface and inject the solver dependency into Mapper.

Required Changes
----------------

1. Introduce a solver interface.

   Example shape:

       type SolverEngine interface {
           ValidateConstraintSet(cs ConstraintSet, coordAfter func(RoomID) (int, int, bool)) error
           ComputeRebuildResult(cs ConstraintSet, enterID, fromID RoomID, dirMoved string) (RebuildResult, error)
       }

   Exact naming is up to you.

2. Make ConstraintSolver implement that interface.

3. Remove direct solver construction from Mapper.

   Eliminate the pattern where Mapper constructs a new solver internally.

4. Inject the solver dependency into Mapper.

   This can be done in a simple conservative way, for example:

   - Mapper has a solver field of interface type, OR
   - Mapper has a small solver factory dependency, if necessary

   Prefer the simplest solution.

5. Preserve solver context updates correctly.

   Because solver depends on current mapper state snapshot, ensure the injected
   solver still receives the correct up-to-date context.

   It is acceptable if the injected dependency is a factory or provider rather
   than a permanently stored solver instance, as long as the architecture is
   cleaner and Mapper no longer constructs concrete solver types directly.

6. Preserve behavior.

   All existing behavior and tests must remain unchanged.

Non-Goals
---------
Do NOT:

- redesign algorithms
- redesign mapper logic
- redesign context schema
- split into packages
- change public behavior intentionally
- perform broad unrelated cleanup

Success Criteria
----------------
This pass is successful if:

1. The code compiles cleanly.
2. Existing tests still pass.
3. Runtime behavior remains unchanged.
4. Mapper no longer directly constructs ConstraintSolver.
5. Mapper depends on a solver abstraction rather than a concrete implementation.

Implementation Guidance
-----------------------
This is a dependency inversion step.

A conservative acceptable end state is:

- introduce a SolverEngine interface
- introduce a solver provider/factory abstraction if needed
- Mapper requests a solver through that abstraction
- concrete ConstraintSolver creation happens outside of Mapper's core logic

Keep this simple and surgical.

Deliverable
-----------
Make the code changes, ensure there are no errors, and preserve behavior.
