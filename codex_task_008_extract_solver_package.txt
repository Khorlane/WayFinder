WAYFINDER CODEX TASK 008
Extract Solver Into Its Own Package (Behavior Preserving)

Objective
---------
Complete the next architectural separation step by moving solver code out of
package main and into its own package.

Current state is strong:

- solver uses explicit solver state
- solver returns explicit rebuild results
- Mapper depends on a SolverEngine abstraction
- solver creation is injected via SolverProvider

Now we want the next clean boundary:

- solver code lives outside package main
- Mapper depends on solver package abstractions
- concrete solver implementation is physically separated from application code

Important
---------
Do NOT change observable behavior.
Do NOT redesign algorithms.
Do NOT change spatial semantics.
Do NOT change discovery semantics.
Do NOT change rebuild semantics.

This is a packaging / structure refactor only.

Why
---
Right now the architecture is logically separated, but solver code still lives
inside package main.

We want physical separation to match architectural separation.

Target direction:

main / Mapper / session / CLI
        │
        ▼
solver package
   - SolverEngine
   - SolverContext
   - SolverRoomState
   - RebuildResult
   - ConstraintSolver

Scope
-----
Create a dedicated solver package and move solver-specific types and logic into it.

Required Changes
----------------

1. Create a new package for solver code.

   Use a simple repo-local package name, for example:

   - solver

   Put the extracted files in a reasonable location.
   Since this repo currently keeps main.go in the project root, keep the move
   simple and readable.

   Example acceptable direction:

   - solver/
       engine.go
       context.go
       rebuild.go

   Exact file split is up to you.
   Keep it modest.

2. Move solver-specific types into the solver package.

   This includes, as appropriate:

   - SolverEngine
   - SolverProvider (if it belongs there)
   - SolverContext
   - SolverRoomState
   - RebuildResult
   - RebuildRoomState
   - ConstraintSolver
   - DefaultSolverProvider

3. Keep Mapper in package main.

   Mapper should import and use the solver package rather than holding local
   copies of solver-specific types.

4. Preserve clean dependency direction.

   Desired dependency flow:

   package main  →  solver package

   Avoid creating circular dependencies.

5. Handle shared types carefully.

   Some types currently used by both Mapper and solver may need to remain in a
   common location or be referenced from main if solver can use them without
   introducing cycles.

   Be conservative here.

   If necessary, introduce a very small shared-types package only if there is
   no cleaner way to avoid cycles.
   Prefer NOT to do that unless required.

6. Preserve behavior.

   Existing tests and runtime behavior must remain unchanged.

Non-Goals
---------
Do NOT:

- redesign algorithms
- redesign the solver API more than needed
- split into many packages
- move main.go out of the project root
- perform broad unrelated cleanup
- change public behavior intentionally

Success Criteria
----------------
This pass is successful if:

1. The code compiles cleanly.
2. Existing tests still pass.
3. Runtime behavior remains unchanged.
4. Solver code no longer lives in package main.
5. Mapper depends on the extracted solver package cleanly.
6. There are no circular imports.

Implementation Guidance
-----------------------
Be practical.

A good outcome is more important than a perfect package taxonomy.

If some narrow shared type aliases or small shared definitions are needed to
avoid cycles, do the minimum necessary.

Keep the refactor surgical and behavior-preserving.

Deliverable
-----------
Make the code changes, ensure there are no errors, and preserve behavior.
