WAYFINDER CODEX TASK 003
Constraint Solver Extraction (Behavior Preserving)

Objective
---------
Take the next architectural step after ConstraintSet extraction.

ConstraintSet is now a meaningful explicit model.
Validation and rebuild both consume it.

This task extracts the solver/validator responsibilities out of Mapper so the
architecture becomes clearer:

- Mapper coordinates
- Constraint builder produces ConstraintSet
- Constraint solver validates and rebuilds layouts

Important
---------
Do NOT change observable behavior.
Do NOT redesign the algorithm in this pass.
Do NOT change spatial semantics.
Do NOT change discovery semantics.
Do NOT rewrite enterIncremental logic beyond what is necessary to call the
extracted solver functions.

Why
---
Right now Mapper still contains too many responsibilities:

- layout state
- lock discovery
- constraint building
- constraint validation
- rebuild/solver logic

The next clean separation is:

1. Mapper owns state and coordination
2. ConstraintSet expresses what must be true
3. Solver code validates / rebuilds from ConstraintSet

Scope
-----
Extract validation and rebuild logic into a dedicated solver module/file.

A new file is preferred if convenient, for example:

- constraint_solver.go

Required Changes
----------------

1. Introduce a solver-oriented abstraction.

   This does NOT need to be a final grand design.
   A simple approach is fine, such as:

   - free functions in a new file, OR
   - a small solver type that operates on mapper state

   The key architectural goal is that validation/rebuild logic no longer lives
   as primary responsibilities inside Mapper.

2. Extract validation logic.

   Move the logic currently centered around:

   - validateConstraintSet
   - validateLockedAdjacencies

   into the solver module or solver type.

   Mapper may keep thin wrapper methods if useful, but the real logic should
   live outside the core Mapper type.

3. Extract rebuild logic.

   Move the logic currently centered around:

   - rebuildDiscoveredLayout

   into the solver module or solver type.

   Mapper should remain the owner of room/occupancy/lock/topology state, but
   the rebuild algorithm itself should no longer be primarily implemented as a
   large Mapper method.

4. Keep Mapper as coordinator.

   After refactor, Mapper should mainly:

   - hold state
   - build ConstraintSet
   - decide when validation/rebuild is needed
   - delegate validation/rebuild to the solver

5. Preserve behavior.

   Existing tests and current runtime behavior must remain unchanged.

Non-Goals
---------
Do NOT:

- redesign the rebuild algorithm
- redesign the validation semantics
- redesign ConstraintSet schema
- remove locks mechanism yet
- change ordered-alignment rules
- change no-between rules
- split into multiple packages
- move main.go out of the project root
- perform broad cleanup unrelated to this task

Success Criteria
----------------
This pass is successful if:

1. The code compiles cleanly.
2. Existing tests still pass.
3. Runtime behavior remains unchanged.
4. Validation logic is no longer primarily implemented inside Mapper.
5. Rebuild logic is no longer primarily implemented inside Mapper.
6. Mapper now reads more clearly as a coordinator + state holder.

Implementation Guidance
-----------------------
This is a structure-only refactor.

Good end state examples:

- Mapper.BuildConstraintSet() remains on Mapper
- Solver.ValidateConstraintSet(...) performs validation
- Solver.RebuildLayout(...) performs rebuild logic
- Mapper.Enter(...) and related flow delegate to solver code

Keep the refactor surgical.
Prefer small wrapper methods over invasive rewrites.

Deliverable
-----------
Make the code changes, ensure there are no errors, and preserve behavior.
