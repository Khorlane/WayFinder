WAYFINDER LLM DEVELOPMENT CONTEXT

WayFinder is a MUD client with an integrated navigation system designed
for a specific known world.

The system builds a readable spatial map of rooms as the player explores.

Important distinction:

World topology and discovery state are authoritative.
The grid layout is a rebuildable projection.

The projection may change over time to maintain readability.

--------------------------------------------------

ARCHITECTURE

WayFinder consists of five components:

WayFinder Client Shell (WCS)
WayFinder Event Gateway (WEG)
WayFinder Navigation Engine (WNE)
WayFinder Map Renderer (WMR)
WayFinder Map Panel (WMP)

Data flow:

WCS → WEG → WNE → WMR → WMP

Responsibilities:

WCS
Host application.

Responsibilities:

- Telnet transport using github.com/reiver/go-telnet
- UI using the Fyne GUI toolkit
- command input
- MUD output display
- event routing

WCS contains no navigation logic.

WEG
Parses MUD output and converts it into navigation events.

Example events:

EnterRoom(roomID)
ExitsSeen(roomID, exits)

WEG contains no mapping logic.

WNE
Core navigation engine.

Responsibilities:

- track discovered rooms
- maintain spatial projection
- enforce layout constraints
- rebuild layout when conflicts occur

WNE contains no UI logic.

WMR
Converts navigation state into renderable structures.

WMP
Displays the map in the UI.

--------------------------------------------------

DISCOVERY MODEL

WayFinder maps only the discovered portion of the world.

Rules:

A room becomes discovered when the player enters it.

If a room is discovered, all of its exits are revealed.

There are no hidden exits in the world design.

Undiscovered rooms are not placed on the map.

--------------------------------------------------

SPATIAL MODEL

WayFinder does NOT use rigid adjacency.

Instead it uses ordered directional constraints.

Examples of valid relationships:

Vertical ordering:

A
.
B

Horizontal ordering:

A ... B

Rules:

Directional ordering must be preserved.

Spacing gaps are allowed.

No room may appear between locked ordered pairs on the same axis.

--------------------------------------------------

LAYOUT STRATEGY

When a new room is discovered:

1. Attempt incremental placement.

2. If constraints remain valid, keep the layout.

3. If constraints are violated, rebuild the layout
   from the discovered subgraph.

The projection may change during rebuild.
This is expected behavior.

--------------------------------------------------

ENGINE INVARIANTS

World topology is immutable.

Discovery state grows monotonically.

The layout projection may change.

Mapping logic must never depend on UI.

Mapping logic must never depend on telnet transport.

--------------------------------------------------

DEVELOPMENT ENVIRONMENT

The current console application is a deterministic
testing harness for the navigation engine.

Movement commands simulate player navigation.

The harness allows developers to verify that the
navigation engine maintains correct spatial relationships.

Future versions will integrate the engine into
a full telnet client with a graphical UI.
